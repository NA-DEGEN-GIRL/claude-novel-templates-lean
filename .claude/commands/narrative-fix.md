서사 리뷰 보고서(`summaries/narrative-review-report.md`)를 기반으로 서사 품질을 수정한다.

## 사용법

`/narrative-fix` 또는 `/narrative-fix C1` 또는 `/narrative-fix C1,H1,H2`

- 인자 없음: 보고서의 전체 수정 항목을 우선순위순 처리
- 항목 지정: 특정 항목만 수정 (예: C1 = CRITICAL 1번)

## 사전 조건

1. `summaries/narrative-review-report.md`가 존재해야 한다. 없으면 `/narrative-review`를 먼저 실행하라고 안내.
2. `CLAUDE.md`를 읽어 소설별 규칙을 확인한다.
3. 보고서의 **"건드리면 안 되는 것"** 목록을 먼저 읽고 숙지한다.

## 수정 절차

수정 항목을 CRITICAL → HIGH → MEDIUM 순서로 처리한다.

### 항목별 수정 프로세스

각 수정 항목에 대해:

1. **보고서 읽기**: 해당 항목의 `문제`, `현재`, `수정 방향`, `적용 범위`, `주의` 필드를 확인
2. **범위 로딩**: 적용 범위의 에피소드 + 전후 1화씩 읽기
3. **"건드리면 안 되는 것" 확인**: 적용 범위 내에 보호 대상이 있는지 체크. 있으면 해당 부분을 피해서 수정.
4. **수정 계획 제시**: 구체적 수정 계획을 사용자에게 보여주고 승인 대기
5. **수정 적용**: 승인 후 `writer.md`의 문체 규칙과 체크리스트를 준수하며 수정
6. **자체 검증**:
   - 수정이 진단된 문제를 실제로 해결하는가?
   - 캐릭터 보이스가 유지되는가? (CLAUDE.md 8절 호칭 매트릭스 참조)
   - 전후 에피소드와 연속성이 깨지지 않았는가?
   - 분량이 목표 범위 이내인가?
7. **요약 갱신**: 수정된 에피소드의 관련 summaries 파일 갱신

### 수정 유형별 전략

**반복 산문 교정** (보고서 "반복 산문 패턴" 섹션):
- 전체가 아니라 과다 반복분만 교체. 핵심 장면의 의도적 반복은 유지.
- 대체 표현은 문맥에 맞는 다양한 감각/동작으로.

**주인공 능동성 복원** (보고서 "주인공 능동성 추이" 섹션):
- 플롯 결과는 유지. HOW만 변경 (정보를 "받는" → "찾아내는").
- 1-2문장의 내적 판단 과정 추가로 능동성 확보.

**정보 덤프 해소** (보고서 CRITICAL/HIGH 중 해당 항목):
- 필수 정보만 추려서 대화/발견/감각으로 분산.
- 2-3화에 걸쳐 분산 가능. 불필요한 정보는 삭제.

**감정 장면 복원** (보고서 "감정 클라이맥스 평가" 섹션):
- 기술/상태 업데이트를 인접 화로 이동.
- 감정 비트에 500자+ 집중 공간 확보.

**복선 삽입** (보고서 CRITICAL/HIGH 중 해당 항목):
- 3-4개 자연스러운 삽입 지점에 1-2문장씩.
- 첫 독에는 놓치고, 재독 시 "아!" 할 정도의 미묘함.

## 수정 로그

처리 완료 후 `summaries/narrative-fix-log.md`에 기록:

```markdown
# 서사 수정 로그

> 수정일: {날짜}
> 기반 보고서: narrative-review-report.md

## 수정 내역

| # | 우선순위 | 항목 | 적용 범위 | 상태 |
|---|---------|------|----------|------|
| C1 | CRITICAL | {제목} | {화수} | ✅ 완료 / ⚠️ 부분 적용 / ❌ 보류 |

## 보류 항목

| # | 보류 사유 |
|---|----------|

## 수정 후 확인 필요

- {수정으로 인해 다른 부분에 영향이 갈 수 있는 항목}
```

## 커밋

- 수정된 본문 + 요약 + fix-log를 함께 커밋
- 메시지: `{소설명} 서사 수정 반영 ({수정_건수}건)`

## 수정 원칙

1. **"건드리면 안 되는 것"은 절대 수정하지 않는다**
2. **플롯 결과는 유지** — HOW를 고치지 WHAT을 고치지 않는다. 구조 변경이 필요하면 사용자에게 먼저 확인.
3. **writer.md 규칙 준수** — 문체, 체크리스트, CLAUDE.md 금지사항 모두 적용
4. **한 항목씩 순차 처리** — 동시 수정 금지. 완료 후 다음 항목으로.
5. **사용자 승인 후 실행** — 수정 계획을 먼저 보여주고 승인 받기
